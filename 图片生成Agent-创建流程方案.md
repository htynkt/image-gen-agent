## 〇、流程总原则（为什么这么排顺序）

因为定位是**工具型 Agent**，所以"灵魂"不是对话，而是**工具调度内核**。流程必须围绕它：

1. **调度内核优先**：先建"工具调度器"（Tool Use 引擎），再造具体工具。骨架对了，加工具像插卡一样简单。
2. **每个工具都能独立验收**：不憋大招，做完一个测一个。
3. **本地算法工具先行验证链路**：拼豆图不花钱、完全可控，用它第一个打通"对话→调度→出图"全链路。
4. **个性化后置**：先把功能跑通，再加语气人设和记忆，避免一上来就过度耦合。

> ⚠️ 与直觉不同：很多人会先搭漂亮对话界面，但工具型 Agent 应该**先搭调度内核**。这是重新定位后的核心调整。

### Step 2　工具调度内核 ⭐⏱️ 1 天（最重要）

**🎯 目标**：建立 **Tool Use 引擎**——统一的"工具接口 + 注册表 + 调用循环"。这是整个 Agent 的灵魂。

**🔧 关键动作**

① 定义统一工具接口（所有工具长一个样）：
```python
@dataclass
class Tool:
    name: str
    description: str          # 给 LLM 看，决定何时调用
    parameters: dict          # JSON Schema，告诉 LLM 怎么传参
    handler: Callable         # 真正执行的函数
```

② 建注册表（工具像插卡一样注册进去）：
```python
TOOL_REGISTRY: dict[str, Tool] = {}
def register(tool: Tool): TOOL_REGISTRY[tool.name] = tool
```

③ 建简化版 ReAct 调度循环（理解→调用→反馈）：
```python
def agent_loop(user_input, history):
    messages = build_system_prompt(profile, memories) + history + user_input
    while True:
        resp = llm.chat(messages, tools=list(TOOL_REGISTRY.values()))
        if not resp.tool_calls:        # 没要调工具 → 最终回复
            return resp.content
        for call in resp.tool_calls:   # 要调工具 → 执行 → 结果喂回去
            result = TOOL_REGISTRY[call.name].handler(**call.args)
            messages.append(tool_result(result))
    # 通常 1~2 轮即结束，这就是"简化版 ReAct"
```

**📦 产物**：`tool_registry.py` + `agent_loop.py`，先用 1 个假工具（返回固定图片）测通。
**✅ 验收**：跟它说"测试一下"，它能触发假工具并返回结果——**调度链路通了**。
**🏷️ 定位**：**Tool Use 模式的核心实现**，这一步做对，后面全顺。
**⚠️ 关键**：工具接口要设计得通用，三个生图能力日后都套这个壳，不要各自为政。

---

### Step 3　第一个工具：拼豆图 ⭐⏱️ 1.5 天

**🎯 目标**：实现真实工具，**零成本打通"对话→调度→出图"全链路**。

**🔧 关键动作**
- `skills/bead_art.py`，按接口注册为 `generate_bead_art` 工具：
  1. 载入 `bead_palette.json` 色板
  2. 缩放到目标格子数（默认 29×29）
  3. **CIEDE2000 色差匹配**每个像素到最接近的豆色
  4. 渲染拼豆效果图（圆形豆子）+ 拼装对照图（网格+色号）
  5. 统计材料清单（每色多少颗）

**📦 产物**：输入图/关键词 → 输出拼豆三件套（效果图 + 对照图 + 材料清单）。
**✅ 验收**：发张照片说"做拼豆"，收到像样的拼豆图和材料表。
**🏷️ 定位**：验证 Tool Use 链路真实可用；本地算法，不烧钱，最适合第一个做。

---

### Step 4　对话循环 + Gradio 界面　⏱️ 1 天

**🎯 目标**：把 Step 2 的内核接到界面，让用户能**自然聊天 + 触发工具**。

**🔧 关键动作**
- `app.py`：Gradio 聊天框 + 图片上传 + 图片展示区
- 接通 `agent_loop`，历史对话维护
- 图片类结果在界面上展示出来（拼豆图直接显示）

**📦 产物**：一个真能用的聊天界面，发图/说话能触发拼豆出图。
**✅ 验收**：完整体验——你打字/发图 → 它回话 + 给图。
**🏷️ 定位**：简化版 ReAct 的落地（一轮对话完成）。

---

### Step 5　个性化档案注入　⏱️ 0.5 天

**🎯 目标**：让它"是你的 Agent"——按你的语气和风格说话。

**🔧 关键动作**
- 填 `config/profile.yaml`（称呼/语气/默认风格）
- `build_system_prompt()` 读取并注入系统提示词
- 生图时默认风格自动套用（如拼豆默认尺寸、AIGC 默认画风）

**📦 产物**：Agent 开始叫你指定称呼、用你设定的语气。
**✅ 验收**：同样的话，换不同 profile 文案，回复语气明显不同。
**🏷️ 定位**：Tool Use 之上的个性化层（静态档案）。

---

### Step 6　扩展工具：AIGC 创意图 + 人像转卡通　⏱️ 2.5 天

**🎯 目标**：把另外两个生图能力**套同样的工具接口**插进注册表，验证内核的可扩展性。

**🔧 关键动作**
- `skills/aigc_creative.py` → `generate_aigc_creative`：
  关键词 → 提示词增强（套默认风格）→ 文生图 API → 出图
- `skills/portrait_toon.py` → `portrait_to_cartoon`：
  人像 → （可选人脸检测/抠图）→ 图生图 API（盲盒风提示词模板）→ 多张候选
- ⚠️ 接好 API 成本控制：每日调用上限、失败重试限次

**📦 产物**：三种图都能从同一个对话入口生成。
**✅ 验收**：三种输入各自触发对应工具，无需改 `agent_loop`（证明内核设计正确）。
**🏷️ 定位**：**Tool Use 可扩展性的真正检验**——加工具不该动核心代码。

---

### Step 7　长期记忆系统　⏱️ 1 天

**🎯 目标**：让它记住你的偏好和过往作品，越用越懂你。

**🔧 关键动作**
- `core/memory.py`：
  - 每轮结束后用轻量 LLM 抽取"值得记住的事实"→ 写入 `memories.jsonl`
  - 下一轮按话题检索 Top-K 相关记忆注入提示词（MVP 用关键词匹配，进阶上向量检索）
- 作品库：`data/outputs/` 记录历史生成图

**📦 产物**：它能引用"上次那个""你喜欢的风格"。
**✅ 验收**：告诉它一个偏好，几轮后它能在相关场景主动用上。
**🏷️ 定位**：个性化层（动态记忆），Tool Use 之外的能力增强。

---

### Step 8　整合打磨　⏱️ 1 天

**🎯 目标**：把零散功能打磨成顺滑的完整体验。

**🔧 关键动作**
- 意图路由消歧（含糊输入会反问，不硬猜）
- 错误兜底（API 失败、图片格式不对等）
- 历史作品画廊（本地浏览过往生成图）
- 个性化语气润色

**📦 产物**：完整可用的本地原型。
**✅ 验收**：连续对话 20 轮，体验流畅、不出明显错。
**🏷️ 定位**：收尾，提升完成度。

## 五、关键技术决策点（基于 Tool Use 定位）

| 决策点 | 推荐方案 | 理由 |
|--------|----------|------|
| 工具接口 | 统一 `Tool(name, desc, parameters, handler)` | 三个工具长一样，内核不挑食 |
| 工具调度 | LLM 原生 Function Calling | 不自己写解析，靠模型决定调哪个 |
| ReAct 循环深度 | 限制 1~2 轮即终止 | 你的任务一轮就能完成，别上复杂多轮 |
| 意图不明时 | 让 Agent 反问而非硬猜 | 工具型 Agent 宁可确认，不要错调 |
| 加新工具 | 只写 `skills/xxx.py` + 注册，不动 `agent_loop` | 这是 Tool Use 内核设计正确的标志 |

## 六、风险与回退

| 风险 | 触发 | 回退方案 |
|------|------|----------|
| Function Calling 不准 | 总调错工具 | 加关键词规则兜底路由 |
| 拼豆颜色失真 | 肤色/渐变跑偏 | CIEDE2000 → 再加抖动（dithering） |
| 生图 API 不稳 | 偶发失败/超时 | 重试 2 次 + 友好提示，不卡死 |
| API 烧钱 | 反复试图 | 每日调用上限 + 相同请求缓存 |
| 人脸隐私 | 照片泄露 | 人像不落库，仅传 API |
