# 项目架构详解（面试讲解版）

> 目标：看完这份，你能在面试时把「项目架构、前后端与 Agent 怎么调用、核心机制、设计决策」讲得清清楚楚。
> 配套代码仓库：https://github.com/htynkt/image-gen-agent

---

## 一、一句话介绍这个项目

> 这是一个**个性化图片生成 Agent**：用户在网页里发文字或图片，AI 会调用工具生成**拼豆图纸**或 **AIGC 创意图**，还能多模态看图对话。架构上采用 **Vue3 前端 + FastAPI 后端 + Agent 核心**的三层分离结构。

---

## 二、整体架构（先记这张图）

```
┌─────────────────────────────────────────────────────────────┐
│                      浏览器（用户）                           │
│              Vue3 聊天界面（发文字 + 图片）                   │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP POST（同步）
                           │ /api/chat  { text, image(base64) }
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              后端：FastAPI（backend/main.py）                │
│   ① 图片 base64 落盘到 data/uploads/                         │
│   ② 调用 agent_loop(text, image_path)                        │
│   ③ 从回复里正则抽出图片路径 → 转成 /files URL               │
│   ④ 返回 { reply, images:[...] }                             │
└──────────────────────────┬──────────────────────────────────┘
                           │ 函数调用（同进程，非网络）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              Agent 核心（agent.py：agent_loop）               │
│   组装消息 → while 循环调 LLM → LLM 决定调工具就执行 → 回复   │
│                                                              │
│   工具（skills/）：                                           │
│     ├─ generate_bead_art  拼豆图（本地算法，零成本）          │
│     └─ generate_aigc      AIGC 创意图（调文生图 API）         │
│                                                              │
│   能力（core/）：                                             │
│     ├─ memory.py   个性化档案 + 对话历史 + 上下文压缩         │
│     └─ logger.py   日志                                       │
└──────────────────────────┬──────────────────────────────────┘
                           │ OpenAI 兼容 API
                           ▼
                    大模型（gpt-4o 多模态 + 文生图）
```

**三层职责一句话总结：**
- **前端**：负责交互（输入、上传、显示文字和图片），不碰业务逻辑
- **后端 FastAPI**：HTTP 网关 + 图片落盘 + 调用 Agent + 把结果转成前端能用的格式
- **Agent 核心**：真正的「大脑」，决定调什么工具、怎么回复、记什么

## 三、一次请求的完整调用链（⭐ 面试核心，一定要会讲）

以「用户上传一张图 + 说『把这个做成拼豆』」为例，完整走一遍：

### 第 1 步：前端（Vue）
- **文件**：`frontend/src/components/ChatWindow.vue`
- 用户点发送 → `ChatWindow` 调 `api.js` 的 `sendMessage(text, imageFile)`
- **`frontend/src/api.js`**：用 `FileReader` 把图片读成 **base64**，组成请求体：
  
  ```json
  { "text": "把这个做成拼豆", "image": "data:image/jpeg;base64,/9j/..." }
  ```
- 通过 `fetch('/api/chat')` 发出。注意：前端写的是**相对路径** `/api/chat`，由 **Vite 的 proxy**（`vite.config.js`）转发到后端 `localhost:8000`，**开发期免跨域**。

### 第 2 步：后端（FastAPI）
- **文件**：`backend/main.py` 的 `@app.post("/api/chat")`
- 做三件事：
  1. **图片落盘**：把 base64 解码，存到 `data/uploads/<uuid>.png`，得到本地路径
  2. **调用 Agent**：`reply = agent_loop(text, user_image=图片路径)`
  3. **抽取图片**：用正则从 `reply` 文本里抠出 Agent 生成的图片**绝对路径**，转成前端可访问的 `/files/outputs/xxx.png` URL
- 返回：`{ "reply": "茜茜～拼豆图纸好啦...", "images": ["/files/outputs/bead_pattern_xxx.png"] }`

> 🔑 **关键点**：后端和 Agent 是**同一个进程里的函数调用**（不是网络调用）。`backend/main.py` 通过 `sys.path` 把项目根加进来，直接 `from agent import agent_loop`。后端只是给 Agent **包了一层 HTTP**。

### 第 3 步：Agent 核心（agent.py 的 agent_loop）
这是最核心的部分，**内部是一个 while 循环**：

```python
def agent_loop(user_text, user_image=None):
    # 1. 组装消息
    profile = load_profile()                              # 读个性化档案
    messages = [ system提示(profile) ] + 历史 + [ 本次输入 ]
    # 如果有图片，把图片转成 base64 塞进 user 消息（多模态）

    # 2. while 循环：反复问 LLM，直到它给出最终回复
    while step < max_steps:
        resp = chat_with_retry(... messages, tools=工具表 ...)
        if LLM 想调工具:
            执行工具（拼豆/AIGC）→ 把结果塞回 messages → 继续循环
        else:
            return LLM 的最终回复    # 循环结束
```

具体到「做成拼豆」这个例子：
1. LLM 看到「图片 + 做成拼豆」→ 决定调用 `generate_bead_art(image_path=...)`
2. 后端执行拼豆算法 → 图片存到 `data/outputs/bead_pattern_xxx.png` → 返回「已生成：D:\\...\\bead_pattern_xxx.png」
3. 把这个结果**塞回 messages**，再问 LLM 一次
4. LLM 看到工具成功了 → 用温柔语气回复用户 → 循环结束

### 第 4 步：回到前端
- `ChatWindow` 拿到 `{reply, images}`
- 把回复文字显示成 Agent 气泡，`images` 数组渲染成**缩略图**（可点击放大、可下载）

## 四、Agent 的五大核心机制（面试必考）

### 1. Agent Loop（核心运行机制）
- **本质**：一个 `while` 循环
- **为什么用循环**：LLM 一次调用可能只是「我要调工具」，调完工具要把结果喂回去再问，可能多轮，所以要循环
- **停止条件**（两个）：
  - ① LLM 不再调工具，给出最终答案（正常结束）
  - ② 达到 `max_steps`（防死循环），做一次收尾
- 代码在 `agent.py` 的 `agent_loop`

### 2. Tool Use（工具系统）
- **一个工具 = 函数 + 说明书**：
  - 函数：真正干活（`generate_bead_art` / `generate_aigc`）
  - 说明书：JSON schema，告诉 LLM「叫什么、何时用、传什么参数」
- **注册机制**：`agent.py` 里 `TOOLS`（说明书列表）+ `TOOL_FUNCTIONS`（名字→函数映射）
- **扩展性**：加新工具 = 写一个 skill + 注册两行，**Agent Loop 一行不用改**

### 3. 多模态（图片怎么给 LLM）
- 用户上传的图片，在 `agent_loop` 里用 `_image_to_data_url()` 转成 `data:image/...;base64,...`
- 塞进 user 消息的 `content`（一个数组，含文字 + image_url）：
  ```python
  {"role":"user","content":[
      {"type":"text","text":"做成拼豆"},
      {"type":"image_url","image_url":{"url":"data:image/png;base64,..."}}
  ]}
  ```
- gpt-4o 支持视觉，能真的「看懂」图

### 4. Memory（记忆，分两层）
- **个性化档案**（`config/profile.yaml`）：称呼、语气、默认风格 → 每次都拼进 system 提示词
- **对话历史**（`data/chat_history.json`）：**默认不灌进上下文**（避免回复混乱）；用户说「上次/之前」等词时，才加载历史（`_mentions_history` 判断）
- **上下文压缩**：历史太长时，用 LLM 把旧消息总结成一段摘要，替换掉（`compress_history_if_needed`）

### 5. 健壮性（生产级考量）
- **连接重试**：`chat_with_retry`（对话）、`_generate_with_retry`（文生图），网络抖动自动重连
- **错误兜底**：工具执行用 try-except，报错也不崩，错误信息返回给 LLM 让它自己处理
- **护栏**：`max_steps` 最大轮数，防死循环
- **日志**：关键事件写 `data/agent.log`，事后能查

## 五、模块依赖关系（谁调用谁）

```
backend/main.py
   └── import → agent.py (agent_loop)
                  ├── import → skills/bead_art.py     (generate_bead_art)
                  ├── import → skills/aigc_creative.py (generate_aigc)
                  ├── import → core/memory.py          (profile/历史/压缩)
                  └── import → core/logger.py          (日志)

frontend/src/components/ChatWindow.vue
   ├── import → frontend/src/api.js                    (发请求)
   └── import → frontend/src/components/MessageBubble.vue (消息气泡)
```

**关键设计：Agent 核心（agent.py + skills + core）完全不知道有 Web 的存在**。它能被命令行调用、也能被 FastAPI 调用——**业务逻辑和传输层解耦**。

## 六、关键设计决策（面试加分点）

| 决策 | 为什么这么做 |
|------|------------|
| **前后端分离** | 前端可独立迭代、后端可复用（同一套 API 能接小程序/App）、更接近真实生产 |
| **后端不改 agent_loop，只包一层 HTTP** | 业务逻辑零侵入，agent_loop 还能被命令行/其他入口复用 |
| **图片用 base64 传输** | 前端一个 JSON 搞定，不用处理 multipart 表单，新手友好 |
| **正则从回复里抽图片路径** | 工具返回的是「自然语言里嵌路径」，后端把它结构化成 `images` 数组，前端只管显示 |
| **默认不灌历史进上下文** | 图片生成是「指令式」任务，全量历史会让回复混乱；引用时再调取，兼顾干净和记忆 |
| **存历史时清理图片 base64** | 图片 base64 很大，存进历史会撑爆上下文，所以只留文字 |
| **Vite proxy 代理 /api** | 开发期前端 5173 访问后端 8000 不跨域；生产期前后端同源也不跨域 |

## 七、面试高频问题 + 回答要点

### Q1：介绍一下你这个项目的架构？
**答**：三层架构。前端 Vue3 负责交互，后端 FastAPI 做 HTTP 网关，核心是 Agent。后端通过函数调用（同进程）调用 Agent 的 `agent_loop`，Agent 再调大模型和工具。前端和后端用 HTTP + base64 传图片通信。

### Q2：Agent 是怎么工作的？
**答**：核心是一个 while 循环——把用户输入和可用工具给 LLM，LLM 如果要调工具就执行、把结果喂回去再问，直到它给出最终回复。这叫 ReAct 模式的简化版。

### Q3：工具是怎么调用的？
**答**：用 OpenAI 的 Function Calling。每个工具有「说明书」（JSON schema），LLM 自己决定调哪个、传什么参数。我在代码里维护一个「工具名→函数」的映射表来真正执行。

### Q4：用户发图片，AI 怎么看懂？
**答**：多模态。前端把图片转 base64，后端落盘后，Agent 把图片转成 data URL 塞进消息的 `image_url` 字段，gpt-4o 支持视觉，能直接看图。

### Q5：记忆系统怎么做的？
**答**：分两层。静态的是个性化档案（称呼/语气），每次注入 system 提示词；动态的是对话历史，存 JSON 文件。我做了个策略：默认不把历史塞进上下文（避免回复乱），用户说「上次/之前」时才调取。历史太长还会自动总结压缩。

### *Q7：前后端怎么通信？跨域怎么解决？*
**答**：同步 POST。开发期用 Vite 的 proxy 把 `/api` 转发到后端 8000，浏览器看到的是同源请求，没有跨域；后端也加了 CORS 中间件双保险。生产期前后端同源（FastAPI 挂载前端 dist）也没跨域。

### Q8：遇到过什么难点？怎么解决？
**答（可挑 2-3 个讲）**：
- **API 网络不稳**：给对话和文生图都加了自动重试（指数退避）
- **长对话撑爆上下文**：做了上下文压缩，旧消息让 LLM 总结成摘要
- **多模态历史太大**：存历史时清理掉图片 base64，只留文字
- **前端响应式 bug**：Vue3 里直接改 push 进数组的对象属性不触发更新，改成索引整体替换
- **Agent 输出的图片路径要给前端显示**：用正则从回复文本抽取路径，转成静态文件 URL
